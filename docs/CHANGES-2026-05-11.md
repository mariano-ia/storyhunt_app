## What we shipped (commits)

### Tracking race-condition fix (both repos)

`StoryHuntABM` `b63c495` — feat(start): animated red breathing bg + chat preview + hero card
`StoryHuntWeb` `f3449c7` — fix(checkout): flush add_payment_info event before Stripe redirect

**Bug:** `trackAddPaymentInfo` fired Meta Pixel + GA4 + PostHog right before
`window.location.href = stripe_session.url`. All three SDKs queue async XHRs —
the browser navigated away before the requests flushed, dropping the event for
most users. Mathematical proof it was a race: last week reported 1 add_payment_info
event but 3 purchases — purchases fire server-side (Stripe webhook, no race) so
they always land, while add_payment_info fires client-side and almost never did.

**Fix in `src/lib/analytics.ts`:**
- GA4 events now use `transport_type: 'beacon'` always (survives pagehide).
- `track()` accepts `{ flush: true }` option → PostHog `send_instantly: true`.
- New `flushPendingEvents()` helper awaits 300ms so Meta Pixel (no flush API)
  has time to fire its XHR before navigation.

**Fix in `StoryHuntWeb/main.js:471`:** same race in the home `/` checkout flow.
PostHog capture now uses `{ send_instantly: true }` and we `await` 300ms before
`window.location.href`. Mirrors the ABM fix per the dual-surface rule.

**Expected next-week impact:** ratio `begin_checkout → add_payment_info` jumps from
6.7% (broken) back to 50–70% (real funnel). If after the fix it stays under 20%,
the problem is real checkout friction, not measurement.

### `/start` hero redesign (ABM only — home `/` keeps its own hero)

Same commit `b63c495`. Visual hierarchy was the diagnosis: 7 elements competing
in the previous hero (logo, H1, chat, 3 steps, CTA, price, refund) — the chat
preview and the 3 numbered steps were both explaining "what the product is" so
neither landed.

- **Single hero card** wrapping H1 + chat + CTA + price + refund. Internal subtle
  gradient + 1px animated gradient border. Card has its own breathing rhythm
  (22s for the inner gradient, 18s for the border, opposite directions).
- **ChatPreview component (`src/app/start/page.tsx`):** two typing bubbles appear
  in parallel ("as if both sides are typing"), resolve staggered to text, then a
  third bubble drops in. Total ~2.4s, plays once on mount, static after. Respects
  `prefers-reduced-motion`.
- **Chat copy:**
  - "Did you know 21 elephants once walked across the Brooklyn Bridge?"
  - "wait, elephants??"
  - "1883 opening day. P.T. Barnum led them to prove it wouldn't collapse. You're standing on the same stones."
  Real historical fact (P.T. Barnum did march 21 elephants over the bridge on
  opening day to prove its stability). Anti-tourist voice, payoff-first.
- **3 steps demoted** to a slim horizontal spec strip below the card:
  `01 WALK · 02 CHAT · 03 DECODE`. Still present but visually subordinated.
- **Section background:** 2 blurred red blobs (scale 1→1.28, opacity 0.55→1)
  breathing in offset (10s and 12s cycles — one inhales while the other
  exhales). Cyan blob removed per user feedback ("solo rojo, más respiración").

### Conversion-review pipeline enhancement

Same commit (`b63c495` initial fix, follow-up commit pending in this session).

`analyze.py` now reads the last 6 `docs/CHANGES-YYYY-MM-DD.md` files and threads
them into the user_message sent to Opus 4.7. Previously the LLM only saw this
week's PostHog data + the last 4 weeks of proposals from `state.json` —
attributing deltas to specific deploys required inference. With the deploy log
in scope, the next analysis (Mon 2026-05-18) will be able to say "ratio X
jumped because the tracking race was fixed on 2026-05-11" rather than guess.

The `SYSTEM_PROMPT` was updated to teach the model to (a) attribute deltas to
deploys when timing matches, and (b) treat prior proposals that appear in a
later CHANGES file as "implemented" and evaluate their impact in retrospect.

  `main.js:35-57`). Dual-surface rule applies to checkout/funnel logic
  (mirrored) but not to marketing hero — they are intentionally different
  surfaces.
